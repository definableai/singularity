## CLI: `sg` (the complete command registry)

The entrypoint is **`sg`**, code in **`src/cli/`** — not `src/scripts/`, which contains
only BaseScript files (08's strict discovery walks that directory; framework CLI modules
inside it would either error the walker or need a skip-list that violates strict-loudness).
The old name "scripts" meant two things; `sg` means one.

## `sg doctor`

One command answers "why won't it boot": python version, docker services up, ports free,
env vars vs Settings fields (missing / unknown / empty), DB + Redis reachable, alembic
single head + pending migrations, pending/drifted scripts (08), store partitions +
dashboard token sanity (09). Each check prints pass/fail + the fix. Pure aggregation of
checks the framework already performs — surfaced before boot instead of as a stack trace
during it.

## Config sync (template can't drift)

`.env.template` is **generated** from the Settings class (01): field names, types,
defaults, field descriptions as comments. `sg config sync` writes it; `--check` (CI)
fails if the committed template differs. Hand-editing the template is
impossible-by-policy.

## API snapshot gate

`openapi.json` committed in-repo, serialized deterministically (sort_keys). `sg api
snapshot --check` fails CI when the running schema differs; regenerating the snapshot in
the same PR is the explicit, reviewable act — the human reading the snapshot diff IS the
breaking/additive judgment. No breaking-change classifier: that's 200–400 lines
re-implementing oasdiff where a false negative ships a break under a green build. Same
generated-artifact-equality idiom as `sg config sync --check` and the alembic drift
check — one convention, three gates. Pairs with `sg errors export` as the two contract
files frontends consume.

## Tests

- pytest, `asyncio_mode=auto`; DB URL from env (`.env.test`), never hardcoded.
- conftest: schema from `Base.metadata.create_all` (fast), per-test session rollback,
  `app`/`client` fixtures.
- **Obs/trace in tests**: `OBS_TRANSPORTS=["jsonl"]` pointed at pytest tmp for ALL
  events; the failing-test dump helper filters `kind == "journey"` at read time. (One
  pipeline — there is no per-kind transport routing to configure.)
- **Authenticated requests, one line**: `client.as_user("usr_1", claims={...})` /
  `as_api_key(...)` — injects a test auth provider producing that Principal, no token
  minting, no monkeypatching. (Dev-server equivalent: `AUTH_DEV_PRINCIPAL`, owned by 03.)
- Ships with tests for the framework itself (registrar strictness, commit-before-response
  ordering, auth chain, obs pipeline, tracer redaction/scrubbing) + `test_healthz.py` as
  the service-test example.

## CI

Template ships a working GitHub Actions workflow (thin, portable):

- lint (`ruff check` + format check) → typecheck (pyright or mypy — pick one, ship it
  configured) → `pytest` against service-container PG/Redis
- **migration gate**: single alembic head + autogenerate drift check (02)
- **contract gates**: `sg api snapshot --check` + `sg config sync --check`
- **dependency audit**: `pip-audit` (or uv equivalent) on the lockfile; allowlist file
  for accepted risks
- image build on main; deploy step left as a stub — the template documents the contract
  (run `sg db migrate` + `sg script run --pending`, then roll pods)
- `uv.lock` committed; Renovate/Dependabot config included but disabled by default

## Docker / compose

- `Dockerfile`: python3.13 slim, uv, non-root.
- `compose.yaml` (dev): postgres (pgvector), redis (**`appendonly yes`** — with acks_late
  the queue is the durability story, 04), api (`uvicorn --reload`), worker
  (`--pool=threads --no-beat`), beat (single replica). Healthchecks +
  `depends_on: service_healthy`.
- `compose.prod.yaml`: api (`uvicorn --workers N`) + worker + beat only (external managed
  pg/redis).
- Migrations: explicit CI/deploy step, not entrypoint magic.
